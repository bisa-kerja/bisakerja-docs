# Services

The services section is the landing area for service-owned documentation that will later be synchronized from the individual repositories.

The service documentation integration standard defines how this section should work as the bridge between platform-level documentation and service-owned source repositories.

## Integration Model

The integration approach for service documentation is defined in [Service Documentation Integration](../standards/service-documentation-integration.mdx).

At a high level:

- this repository owns the landing pages and cross-service context
- the service repositories own detailed technical content
- synchronized docs will live under each service's `synced/` subtree

## Navigation Boundary

To keep discoverability clean:

- use [Overview](../overview/index.mdx) and [Start Here](../overview/start-here.mdx) for platform understanding and onboarding
- use the `services/` landing pages to choose the correct service boundary
- use `synced/` content only when service-level implementation detail is needed

## Structure Rule

Each service directory contains:

- a locally maintained `index.mdx` page for service summary and navigation
- a `synced/` directory reserved for imported service documentation

## Reading Path

Readers should usually move through the documentation in this order:

1. start with [Overview](../overview/index.mdx) for product and system context
2. open the relevant service landing page for ownership boundaries and expected content
3. read synchronized service docs in `synced/` when implementation-level detail is needed
