# Backend API Observability

This document defines the initial observability model for the Bisakerja Backend API. The backend must make user-facing failures diagnosable without exposing secrets, raw CV contents, raw Model API payloads, or raw external job source payloads.

Observability should support local debugging, staging validation, production monitoring, and future central documentation sync without changing API contracts.

## Observability Goals

- Correlate every API error with a request id.
- Distinguish backend, database, Model API, Scraper/job freshness, email, and upload-storage failures.
- Measure latency for user-facing flows and downstream dependencies.
- Capture business events needed to understand MVP workflows.
- Keep logs structured, searchable, and safe.
- Support incident triage using the platform failure scenario priorities.

## Signals

| Signal          | Purpose                                                        | Initial implementation direction                                        |
| --------------- | -------------------------------------------------------------- | ----------------------------------------------------------------------- |
| Structured logs | Debug requests, failures, auth events, dependency behavior     | JSON logs with request id and redaction                                 |
| Request id      | Correlate frontend reports, logs, errors, and downstream calls | Default header `x-request-id`                                           |
| Error reporting | Group unexpected exceptions and dependency failures            | Future provider integration if a dedicated reporting service is adopted |
| Health checks   | Separate process liveness from dependency readiness            | `/health/live` and `/health/ready`                                      |
| Metrics         | Track latency, throughput, errors, and dependency state        | Start with logger-compatible counters/timers; add metrics backend later |
| Audit events    | Preserve sensitive workflow history without sensitive payloads | Structured audit logs or audit table after implementation decision      |

## Structured Logging

Logs should be machine-readable and consistent across modules.

Required fields for request summary logs:

| Field        | Description                                         |
| ------------ | --------------------------------------------------- |
| `timestamp`  | ISO timestamp generated by logger                   |
| `level`      | Logger level such as `info`, `warn`, or `error`     |
| `service`    | Service name, default `bisakerja-api`               |
| `env`        | `local`, `test`, `staging`, or `production`         |
| `requestId`  | Request correlation id                              |
| `method`     | HTTP method                                         |
| `path`       | Request path without raw secrets                    |
| `statusCode` | HTTP response status                                |
| `durationMs` | Request duration                                    |
| `userId`     | Internal user id only when authenticated and useful |
| `errorCode`  | API error code when request fails                   |

Example:

```json
{
	"level": "info",
	"service": "bisakerja-api",
	"env": "staging",
	"requestId": "req_123",
	"method": "GET",
	"path": "/api/v1/jobs",
	"statusCode": 200,
	"durationMs": 42,
	"userId": null,
	"errorCode": null
}
```

Never log passwords, token values, OTP values, reset tokens, raw CV content, extracted CV text, full Model API payloads, service credentials, or full database connection strings.

The current request summary logger records only request metadata: request id, HTTP method, route path without query string, status code, duration, optional authenticated user id, and mapped API error code. It does not log request bodies, raw query strings, cookies, authorization headers, or response payloads.

## Request ID Propagation

Request id rules:

- Accept the configured request id header from trusted callers when present.
- Generate a request id when the incoming request does not include one.
- Add the request id to response headers.
- Include the request id in error response envelopes as `error.requestId`.
- Include the request id in all request, error, audit, and dependency logs.
- Forward the request id to Model API calls.
- Use request id to connect frontend bug reports to backend logs.

The default request id header is `x-request-id`, matching `docs/environment.md`.

## Error Reporting

Unexpected exceptions should be reported without sensitive payloads.

Error reports should include:

- Request id.
- Route group and endpoint name.
- Error class and safe message.
- API error code and status.
- Dependency name when the error is from PostgreSQL, Model API, email, upload storage, or scraper freshness state.
- Runtime environment.
- Release version or commit SHA when available.

Error reports must not include:

- Request bodies for auth, password reset, CV upload, or AI payload routes.
- Tokens, OTP values, passwords, or service credentials.
- Raw downstream response bodies.
- Stack traces in API responses.

## Health Checks

Health endpoints should live under the health module and remain stable for deployment systems.

Current runtime endpoints:

| Endpoint            | Purpose           | Dependency behavior                                  |
| ------------------- | ----------------- | ---------------------------------------------------- |
| `GET /health/live`  | Process liveness  | Does not require database or downstream services     |
| `GET /health/ready` | Runtime readiness | Checks PostgreSQL and critical startup configuration |

If a future dependency-detail endpoint is introduced, it should be internal-only or return sanitized output and must be documented as a separate runtime contract.

Example readiness response:

```json
{
	"success": true,
	"message": "Service is ready",
	"data": {
		"service": "bisakerja-api",
		"status": "ready",
		"env": "staging",
		"dependencies": {
			"postgresql": "healthy"
		}
	},
	"meta": {
		"requestId": "req_123"
	}
}
```

Readiness should fail when PostgreSQL is unavailable. Model API degradation should be visible but should not necessarily make the whole backend unready if non-AI flows can still operate.

The implemented readiness contract checks PostgreSQL with a lightweight query and the configured `HEALTH_CHECK_TIMEOUT_MS`. A healthy database returns `200` with `postgresql: "healthy"`. An unavailable or timed-out database returns the standard `503 SERVICE_UNAVAILABLE` error envelope with sanitized dependency status and logs a dependency failure summary containing request id, dependency name, operation, latency, and result category.

## Latency And Throughput

Track latency for high-value user workflows and dependency calls.

HTTP metrics:

- Request count by method, route group, status class, and environment.
- Request duration by route group.
- Error count by `error.code`.
- Rate limit count by endpoint group.

Dependency metrics:

- PostgreSQL query latency for important repository calls.
- Model API latency, timeout count, invalid response count, and availability.
- Email send latency and failure count for auth flows.
- Upload validation and storage latency for CV Analyzer.
- Job freshness age based on normalized job metadata or ingestion run records.

Business workflow latency:

| Workflow                   | Latency to watch                                                         |
| -------------------------- | ------------------------------------------------------------------------ |
| Job search                 | Database query and response formatting                                   |
| Job detail                 | Normalized job load with related company and requirements                |
| Auth login                 | Credential lookup, hash verification, and session creation               |
| Application tracker update | Ownership check and transaction                                          |
| AI Job Fit                 | Context load, Model API call, response mapping, optional snapshot write  |
| AI CV Analyzer             | Upload validation, context load, Model API call, optional snapshot write |

## Dependency Health

The backend should distinguish dependency states in logs and health output.

| Dependency       | Healthy signal                                            | Degraded or failed signal                                      |
| ---------------- | --------------------------------------------------------- | -------------------------------------------------------------- |
| PostgreSQL       | Lightweight query succeeds within timeout                 | Readiness fails, database errors increase, migrations mismatch |
| Model API        | Health endpoint or contract call succeeds within timeout  | AI routes return `502` or `503`, timeouts increase             |
| Scraper/job data | Fresh normalized records exist within freshness threshold | Search still works but data freshness is stale                 |
| Email provider   | Verification/reset emails accepted by provider            | Auth email flow fails or queues retry                          |
| Upload storage   | Test write/read or configured path exists                 | CV Analyzer upload fails before model call                     |

Dependency logs should include dependency name, operation, latency, result category, and request id.

## Business Events

Business events help debug product workflows and later support analytics. They must not contain sensitive payloads.

Initial events:

| Event                                     | Trigger                     | Sensitive data rule                                  |
| ----------------------------------------- | --------------------------- | ---------------------------------------------------- |
| `auth.registered`                         | Successful registration     | No plaintext email unless policy allows; no password |
| `auth.login_succeeded`                    | Successful login            | No token values                                      |
| `auth.login_failed`                       | Failed login                | Use failure category, not password details           |
| `profile.updated`                         | User profile updated        | No full profile dump                                 |
| `preferences.viewed`                      | Preferences read            | User id and preference id only                       |
| `preferences.created`                     | Preferences first saved     | Changed field names only                             |
| `preferences.updated`                     | Preferences changed         | Changed field names only                             |
| `preferences.notification_toggle_updated` | Email preference changed    | Changed field names only                             |
| `jobs.search_performed`                   | Job search endpoint called  | Query summary only; no raw source payload            |
| `bookmarks.list_requested`                | User views saved jobs       | User id, filter summary, pagination, and count only  |
| `bookmarks.created`                       | User saves a job            | User id and job id only                              |
| `bookmarks.deleted`                       | User removes a saved job    | User id and job id only                              |
| `bookmarks.duplicate_rejected`            | Duplicate save rejected     | User id and job id only                              |
| `applications.list_requested`             | User views tracker records  | User id, filter summary, pagination, and count only  |
| `applications.created`                    | User creates tracker record | User id, job id, status, and source only             |
| `applications.updated`                    | Tracker metadata changes    | Changed field summary, no private notes              |
| `applications.status_updated`             | Tracker status changes      | Old and new status, no private notes                 |
| `applications.duplicate_rejected`         | Duplicate tracker rejected  | User id and job id only                              |
| `ai_job_fit.requested`                    | Fit analysis starts         | No full model payload                                |
| `ai_job_fit.completed`                    | Fit analysis succeeds       | Fit score and model version only                     |
| `ai_job_fit.failed`                       | Fit analysis fails          | Error code and dependency latency only               |
| `ai_job_fit.persisted`                    | Fit snapshot is stored      | Fit score and model version only                     |
| `ai.cv_analysis_requested`                | CV analysis starts          | File metadata id only, no CV text                    |
| `ai.model_failed`                         | Model dependency fails      | Error category and dependency latency                |

Business events can start as structured logs. A dedicated event table or analytics pipeline should be documented before being added.

The initial audit helper emits structured audit logs with `audit: true`, event name, request id, actor id, resource type, resource id, result, and sanitized metadata. Metadata keys that look like passwords, tokens, OTP values, secrets, credentials, cookies, authorization headers, raw CV content, raw model payloads, raw scraper payloads, or database URLs are redacted before logging.

## Alerting Direction

Production alerting should be introduced before public launch.

Initial alert conditions:

- Backend readiness failure.
- Sustained `5xx` rate above threshold.
- PostgreSQL connectivity failure.
- Authentication failure rate spike.
- Model API timeout or error spike.
- Job freshness exceeds documented threshold.
- CV upload failure spike.
- Error reporting volume spike for a new release.

Alert severity should follow user impact:

1. Core platform unavailable.
2. Authenticated workflows degraded.
3. Fit analysis degraded.
4. Data freshness degraded.

## Incident Triage

Use request ids and dependency states to map incidents quickly.

| Symptom                                         | First checks                                                                      |
| ----------------------------------------------- | --------------------------------------------------------------------------------- |
| Users can browse jobs but fit analysis fails    | Model API health, inference payload contract, model timeout logs                  |
| Search results look stale                       | Ingestion freshness metadata, source connector run state, normalized job counts   |
| Authenticated actions fail but public jobs load | Auth/session validation, ownership filters, database write path                   |
| Most user actions fail                          | Backend readiness, PostgreSQL availability, recent deploy and environment changes |
| CV Analyzer fails only for uploads              | Upload validation, storage path, file size/MIME rules, Model API payload mapping  |

## Local Debugging

Local development should support:

- `LOG_LEVEL=debug` without logging sensitive payloads.
- Request id visible in responses and logs.
- Mock Model API behavior through `MODEL_API_ENABLE_MOCK=true`.
- Fixture-backed job data for search and detail workflows.
- Clear readiness output for PostgreSQL plus route-level logs for optional dependency degradation.

## Release Observability Checklist

Before first deployment:

- Request id middleware is enabled.
- Error handler includes request id in error envelopes.
- Logs are structured and redact sensitive fields.
- Health endpoints exist and are documented.
- PostgreSQL readiness is checked.
- Model API failure is observable without breaking unrelated flows.
- Job freshness is observable through metadata or ingestion records.
- Auth, AI Job Fit, and CV Analyzer audit events are defined.
- Production log level and error reporting configuration are documented.
