# Deployment Overview

This page explains how documentation publishing should work operationally once the centralized sync pipeline exists.

It focuses on the flow of documentation changes from service repositories into this repository, not on application deployment itself.

## Publishing Modes

| Mode                     | Trigger                                         | Expected Outcome                                                |
| ------------------------ | ----------------------------------------------- | --------------------------------------------------------------- |
| Merge publish            | Merge to default branch in a service repository | Update the latest central `synced/` docs for that service       |
| Release publish          | Service release tag or release event            | Create or update a versioned documentation snapshot             |
| Scheduled reconciliation | Central scheduled workflow                      | Verify drift, retry failed syncs, and confirm central integrity |

## Merge Publish Workflow

```text
Service docs change
  -> merge to default branch
    -> service repo builds sync bundle
      -> central repo validation
        -> central repo commit or publish of latest synced docs
          -> full docs build validation
```

### Use Case

- Daily documentation iteration
- Small service-specific corrections
- Keeping latest service docs fresh in the central hub

## Release Publish Workflow

```text
Service release
  -> tagged source ref
    -> service repo builds release bundle
      -> central repo validation
        -> publish versioned snapshot
          -> optional release-note summary
```

### Use Case

- Stable release documentation
- Historical reference for incidents or audits
- Service contract changes tied to a release

## Scheduled Reconciliation Workflow

```text
Central scheduler
  -> inspect expected service sync state
    -> compare latest source refs and central manifests
      -> report drift or missing updates
        -> optionally retry or request corrective action
```

### Use Case

- Detect silent pipeline failures
- Verify that `synced/` docs still match expected latest source state
- Catch missing metadata or broken path assumptions over time

## Publish Safeguards

Before a publish is accepted, the central repository should require:

- Bundle manifest validation
- Service-path validation
- Metadata validation
- Asset reference validation
- Whole-site broken-link and production build validation

## Failure Handling

If a publish attempt fails:

1. Reject the incoming bundle
2. Keep the current central docs state unchanged
3. Expose the failure reason to the owning service team
4. Retry only after the source bundle or mapping rules are corrected

## Operational Ownership

| Area                                 | Primary Owner                                     |
| ------------------------------------ | ------------------------------------------------- |
| Service bundle generation            | Source service team                               |
| Central validation and publish logic | Platform docs maintainer or platform engineering  |
| Drift and reconciliation monitoring  | Platform docs maintainer                          |
| Release snapshot lifecycle           | Source service owner with platform docs oversight |
