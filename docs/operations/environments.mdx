# Environments

This page documents the intended environment topology for Bisakerja at the platform level. It is based on current project assumptions and should be refined when service repositories contribute their deployment-specific documentation.

## Environment Model

| Environment | Purpose                                | Expected Characteristics                                                   |
| ----------- | -------------------------------------- | -------------------------------------------------------------------------- |
| Local       | Developer workflow and feature testing | Fast iteration, local service composition, lower data fidelity             |
| Staging     | Pre-release integration and validation | Production-like service interaction, safer test data, deployment rehearsal |
| Production  | User-facing runtime                    | Stable releases, controlled secrets, monitored services                    |

## Local Environment

The local environment should support:

- Frontend development against a reachable backend
- Backend development with local or shared development database access
- Model API availability for integration testing when fit-scoring work is in scope
- Optional scraper execution or fixture-based data for local search behavior

## Staging Environment

The staging environment should exist to validate:

- Cross-service compatibility before release
- Frontend, backend, and model integration together
- Migration safety and configuration correctness
- Non-production operational behavior such as release candidate testing

## Production Environment

The production environment should prioritize:

- Availability of the Backend API and database path
- Reliability of model-serving for fit analysis
- Controlled scraping or ingestion freshness pipelines
- Secure handling of user account and tracker data

## Environment Topology Assumption

```text
Local:
  developer machine
    -> frontend
    -> backend
    -> optional model service
    -> local or shared database

Staging:
  deployed frontend
    -> deployed backend
      -> staging database
      -> staging model service
      -> staging-friendly ingestion path

Production:
  public frontend
    -> production backend
      -> production database
      -> production model service
      -> production ingestion or refresh pipeline
```
