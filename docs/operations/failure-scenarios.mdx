# Failure Scenarios

This page captures the common cross-service incidents that engineers are likely to debug in Bisakerja.

## Scenario 1: Users Can Browse Jobs But Fit Analysis Fails

### Likely Symptoms

- Search and job detail pages still work
- Score, explanation, or skill gap sections fail or time out

### Most Likely Affected Components

- Backend API to Model API integration
- Model API runtime or artifact load
- Backend request shaping for inference

### First Checks

1. Confirm backend connectivity to the model service
2. Confirm inference payload contract compatibility
3. Confirm current model artifact availability

## Scenario 2: Search Results Look Stale

### Likely Symptoms

- Users see old or missing listings
- Search works technically, but the catalog no longer reflects current openings

### Most Likely Affected Components

- Scraper API
- Ingestion scheduler
- Normalization or upsert pipeline

### First Checks

1. Confirm scrape jobs are still running
2. Confirm external source connectors have not drifted
3. Confirm normalized records are still reaching the database

## Scenario 3: Authenticated Actions Fail But Public Pages Still Load

### Likely Symptoms

- Guests can browse jobs
- Login-required features such as tracker updates or preference edits fail

### Most Likely Affected Components

- Backend API authentication or authorization flow
- Database write path for user-scoped records

### First Checks

1. Confirm session or token validation path
2. Confirm user-ownership checks are passing
3. Confirm database writes for user state are healthy

## Scenario 4: Platform-Wide User Failure

### Likely Symptoms

- Frontend loads partially or not at all
- Most user actions fail together

### Most Likely Affected Components

- Backend API availability
- PostgreSQL availability
- Deployment or environment configuration mismatch

### First Checks

1. Confirm backend liveness with `GET /health/live`
2. Confirm backend readiness and database health with `GET /health/ready`
3. Confirm environment configuration and recent deploy changes

## Failure Prioritization Rule

When triaging incidents, classify them by user impact first:

- Core platform unavailable
- Authenticated workflows degraded
- Fit analysis degraded
- Data freshness degraded

This order helps keep debugging aligned with the most visible user pain.
