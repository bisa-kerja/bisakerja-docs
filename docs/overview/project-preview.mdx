# Project Preview

This page explains Bisakerja from the product and user perspective before discussing internal architecture.

## What A User Should See

For a user, Bisakerja should look like a single workspace for job search strategy:

- Discover jobs from multiple platforms in one place
- Evaluate how well a role matches current skills and preferences
- Understand why the score is high or low
- Identify missing skills and likely next actions
- Track applications and learn from outcomes

## Experience Tiers

The current product definition suggests two usage tiers:

| User State         | Available Experience                                                                       |
| ------------------ | ------------------------------------------------------------------------------------------ |
| Guest user         | Browse and search aggregated job listings with limited analysis                            |
| Authenticated user | Access fit scoring, skill gap analysis, personalized preferences, and application tracking |

## Core Screens Or Product Areas

The overview documentation assumes the platform will expose at least these user-facing areas:

- Home or discovery page
- Job search and filter results
- Job detail page
- Fit score and explanation panel
- Skill gap and recommendation area
- User profile and preference settings
- Application tracker dashboard

## Product Narrative

The expected product loop is:

1. A user discovers or searches for jobs.
2. The user opens a role and checks fit.
3. The system explains strengths, mismatches, and missing skills.
4. The user decides whether to apply now or improve readiness first.
5. The user tracks outcomes and uses that history to refine future strategy.

## Why This Matters For Documentation

This product preview is important because technical readers need to understand the user promise behind the architecture:

- The scraper layer exists to power job discovery
- The backend exists to orchestrate product workflows and persist user state
- The model layer exists to generate scoring and explanations
- The frontend exists to make these capabilities understandable and actionable
