# System Landscape

Bisakerja is organized as a multi-service system with one user-facing application and several backend capabilities around it.

## Major Platform Components

| Component         | Primary Responsibility                                                               | Main Consumers                       |
| ----------------- | ------------------------------------------------------------------------------------ | ------------------------------------ |
| Frontend UI       | User interface for discovery, scoring, profile management, and application tracking  | End users                            |
| Backend API       | Main application backend, orchestration layer, auth, business logic, and persistence | Frontend UI, internal platform flows |
| Scraper API       | Collects and normalizes job data from external job platforms                         | Backend API, database layer          |
| Model API         | Serves job fit scoring, explanations, and recommendation outputs                     | Backend API                          |
| PostgreSQL        | Stores user data, job data, preferences, and tracking records                        | Backend API, data ingestion flows    |
| Documentation Hub | Provides centralized product, architecture, and service documentation                | Engineers and stakeholders           |

## Responsibility Breakdown

### Frontend UI

- Renders the job search and fit-scoring experience
- Captures user preferences and application state
- Consumes backend endpoints rather than calling data or model services directly

### Backend API

- Acts as the primary application control plane
- Manages authentication, authorization, and data access
- Orchestrates calls to the Model API
- Exposes job, profile, preference, and application features to the frontend

### Scraper API

- Integrates with external job listing sources such as Glints, Jobstreet, Kalibrr, and Dealls
- Normalizes incoming job data
- Supports recurring refresh of job listings and related market signals

### Model API

- Serves fit-scoring logic
- Returns explainable outputs for score breakdown
- Supports skill gap analysis and recommendation results

## Supporting Concepts

The core services depend on shared platform capabilities:

- A transactional database for application state and normalized job data
- CI/CD pipelines for deployment and, later, docs synchronization
- Service-specific repositories that remain the source of implementation detail

## Current Architectural Assumption

Until the service repositories are synchronized into this docs hub, the platform should be treated as a hub-and-spoke system where the Backend API is the main orchestrator between the frontend, the data ingestion layer, and the model layer.
