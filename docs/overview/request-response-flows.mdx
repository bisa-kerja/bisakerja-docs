# Request and Response Flows

This page documents the main synchronous flows that users feel immediately in the product.

## Flow 1: Job Search

### Purpose

Return a filtered list of normalized jobs to the frontend.

```text
User -> Frontend UI: enter keyword / filters
Frontend UI -> Backend API: request jobs
Backend API -> PostgreSQL: query normalized jobs
PostgreSQL -> Backend API: result set
Backend API -> Frontend UI: filtered jobs
Frontend UI -> User: render results
```

### Notes

- This flow should work for both guest and authenticated users
- This flow depends on scraper freshness indirectly, but not on live scraper availability
- Responses include normalized job cards with pagination, accepted filters, actual sort metadata, and stale indicators when freshness data is available

## Flow 2: Job Detail

### Purpose

Return a single structured job detail record for user review.

```text
User -> Frontend UI: open job detail
Frontend UI -> Backend API: request job detail
Backend API -> PostgreSQL: load normalized job record
PostgreSQL -> Backend API: job detail data
Backend API -> Frontend UI: job detail response
Frontend UI -> User: render structured detail page
```

### Notes

- This flow should expose enough normalized data for fit analysis
- Frontend should not need to interpret source-platform-specific raw fields
- Missing or hidden records return a stable not-found error instead of exposing source or scraper internals

## Flow 3: Fit Score Request

### Purpose

Evaluate how well a user matches a role.

```text
User -> Frontend UI: open fit analysis
Frontend UI -> Backend API: request fit analysis
Backend API -> PostgreSQL: load user profile, preferences, and job record
Backend API -> Model API: send prepared inference payload
Model API -> Backend API: return fit score and explanation
Backend API -> Frontend UI: formatted fit analysis response
Frontend UI -> User: render score, rationale, and skill gap
```

### Notes

- The backend should remain the only service assembling inference payloads
- The frontend should receive product-friendly output, not raw model internals

## Flow 4: Current-User Profile and Onboarding Update

### Purpose

Persist current-user profile changes and return onboarding context that can be reused by other backend workflows.

```text
User -> Frontend UI: update profile, photo metadata, skills, experience, or education
Frontend UI -> Backend API: submit /api/v1/me update request
Backend API -> PostgreSQL: validate ownership and replace/update records transactionally
PostgreSQL -> Backend API: persisted profile state
Backend API -> Frontend UI: updated profile with onboarding status
Frontend UI -> User: render refreshed profile context
```

### Notes

- The backend must scope all `/api/v1/me` mutations to authenticated user identity.
- Nested lists such as skills, experience, and education use replacement semantics in MVP.
- Onboarding status should be computed centrally from persisted account and preference context.

```text
User -> Frontend UI: mark job as applied or interviewed
Frontend UI -> Backend API: submit tracker update
Backend API -> PostgreSQL: validate and persist state change
PostgreSQL -> Backend API: write confirmation
Backend API -> Frontend UI: updated tracker state
Frontend UI -> User: show new application status
```

### Notes

- This flow should be transactional and user-specific
- Even if fit scoring is unavailable, tracker updates should still work if backend and database are healthy

## Flow 6: Saved Job Bookmark

### Purpose

Persist or remove a user-owned saved-job record for later review.

```text
User -> Frontend UI: save or remove job
Frontend UI -> Backend API: submit bookmark request
Backend API -> PostgreSQL: validate current user and normalized job relationship
PostgreSQL -> Backend API: bookmark write or delete result
Backend API -> Frontend UI: saved-job confirmation or no-content delete response
Frontend UI -> User: update saved state
```

### Notes

- Bookmark routes are authenticated and scoped to the current user.
- Saving a job requires the normalized job listing to exist and be visible.
- Duplicate saves return a stable conflict response instead of creating another row.
- Saved job list responses use the same normalized job card shape as job search responses.

## Synchronous Flow Design Guidance

- Frontend calls should terminate at the backend, not at data or model services directly
- Backend responses should be shaped for product use, not internal-only debugging
- Failure states should distinguish between unavailable data, unavailable inference, and unavailable persistence
