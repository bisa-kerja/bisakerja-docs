# Asynchronous Workflows

Not every important system behavior happens on a user request. This page documents the asynchronous workflows that keep the platform useful over time.

## Workflow 1: Job Ingestion Refresh

### Purpose

Collect fresh job data from external platforms on a recurring schedule.

```text
Scheduler -> Scraper API: trigger scrape run
Scraper API -> External Job Platforms: fetch listings
External Platforms -> Scraper API: raw job payloads
Scraper API -> Scraper API: normalize and validate records
Scraper API -> PostgreSQL: upsert normalized jobs
PostgreSQL -> Backend API: fresh data becomes available to search flows
```

### Why It Matters

- Users rely on the platform to surface current openings
- Search quality degrades if this workflow stalls

## Workflow 2: Data Normalization And Enrichment

### Purpose

Convert inconsistent source data into a stable platform format suitable for search and recommendation.

```text
Raw source fields -> normalization rules
Normalization rules -> stable job attributes
Stable job attributes -> database records
Database records -> backend filtering and model input preparation
```

### Typical Concerns

- Platform-specific field names
- Inconsistent salary formatting
- Missing location or skill data
- Duplicate or stale job entries

## Workflow 3: Model Artifact Usage

### Purpose

Keep inference serving stable by loading approved model artifacts rather than recalculating or retraining on request paths.

```text
Approved model artifact -> Model API runtime
Backend prepared payload -> Model API
Model API -> inference result
Inference result -> Backend API -> Frontend UI
```

### Notes

- Retraining is not treated as a user-facing runtime flow here
- The central docs currently focus on inference-serving behavior rather than training pipelines

## Workflow 4: Documentation Sync Readiness

### Purpose

Prepare service-owned docs to be published into this repository through CI/CD in subsequent milestones.

```text
Service repo docs update
  -> metadata validation
  -> path mapping validation
  -> asset collision checks
  -> central sync target
```

### Notes

- This workflow is documented here because stale documentation is also an operational risk
- The full CI/CD design remains a subsequent-milestone concern

## Asynchronous Risk Areas

| Workflow               | Main Risk                             | User Impact                                  |
| ---------------------- | ------------------------------------- | -------------------------------------------- |
| Job ingestion refresh  | external source changes or throttling | stale search results                         |
| Normalization          | parser drift or malformed fields      | degraded filtering and weak model input      |
| Model artifact loading | wrong artifact or failed deployment   | scoring unavailable or inconsistent          |
| Docs sync readiness    | bad metadata or path collisions       | central docs drift or publish failures later |
