# Database Overview

The project brief identifies PostgreSQL as the primary relational database for the Bisakerja platform.

## Data Ownership Model

At the platform level, the database should be treated as the source of truth for user-facing application state and normalized job data.

## Major Data Domains

| Data Domain                 | Description                                          | Primary Writing Service | Primary Reading Service            |
| --------------------------- | ---------------------------------------------------- | ----------------------- | ---------------------------------- |
| User accounts               | Authentication identity and basic account state      | Backend API             | Backend API                        |
| User profiles               | Career profile, experience, and self-declared skills | Backend API             | Backend API, Model API via backend |
| User preferences            | Desired location, role, salary, and work preferences | Backend API             | Backend API, Model API via backend |
| Job listings                | Normalized records from external job sources         | Scraper API             | Backend API                        |
| Job metadata                | Required skills, job attributes, source details      | Scraper API             | Backend API, Model API via backend |
| Application tracker records | User-managed job application state and updates       | Backend API             | Backend API                        |
| Recommendation history      | Stored scoring or explanation snapshots if persisted | Backend API             | Backend API                        |

## Ownership Boundaries

### Backend API-owned records

The Backend API should be considered the owner of:

- Users
- Profiles
- Preferences
- Bookmarks or saved jobs if implemented
- Application tracker state
- Any persisted scoring history presented to users

### Scraper API-owned records

The Scraper API should be considered the owner of:

- Ingested raw job data before normalization
- Normalized job listing refresh logic
- Source freshness and ingestion metadata

### Model API data relationship

The Model API should not be treated as the system of record for transactional data. It consumes prepared data and returns derived outputs such as:

- Fit scores
- Explanation breakdowns
- Skill gap results
- Strategy recommendations

## Data Design Guidance

- Normalized job records should be stable enough for frontend display and backend filtering
- User state should remain transactional and auditable
- Derived AI output should be stored only when it improves the user workflow or historical analysis
- Backend API schema implementation should keep user-owned data, normalized job records, tracker records, and persisted AI snapshots in explicit relational tables with documented ownership and retention rules
- Seed data should be deterministic, synthetic, and normalized; raw source payloads, secrets, raw CV content, and raw model payloads should not be seeded

## Documentation Gap To Fill Later

The central docs define ownership and data domains. Service repositories should provide and synchronize:

- Concrete schema documentation
- Table-level relationships
- Migration strategy
- Data retention policy
