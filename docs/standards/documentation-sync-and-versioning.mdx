# Documentation Sync and Versioning

This standard defines how service-owned documentation is synchronized into the centralized repository and how those synchronized docs are versioned over time.

It extends [Service Documentation Integration](./service-documentation-integration.mdx) by defining the automation model, validation rules, conflict handling, rollback behavior, and release semantics.

## Synchronization Model

Bisakerja should use a **hybrid synchronization model**.

### Why Hybrid

- push-based sync from service repositories keeps the central docs fresh when teams merge documentation changes
- scheduled pull or reconciliation from the central repository detects drift, retries failed syncs, and verifies that expected docs are still aligned
- release-triggered sync allows stable documentation snapshots to be captured alongside service releases

## Sync Triggers

| Trigger                  | Initiator          | Main Purpose                                              |
| ------------------------ | ------------------ | --------------------------------------------------------- |
| Merge sync               | Service repository | Publish latest docs after merge to default branch         |
| Release sync             | Service repository | Capture release-specific documentation snapshot           |
| Scheduled reconciliation | Central repository | Detect drift, retry failed syncs, and verify latest state |

## Folder Mapping

The central repository keeps stable paths even if repository internals evolve.

| Source Repository | Source Root | Latest Sync Target                    | Release Snapshot Target                            |
| ----------------- | ----------- | ------------------------------------- | -------------------------------------------------- |
| `frontend-ui`     | `/docs/**`  | `docs/services/frontend-ui/synced/**` | `docs/services/frontend-ui/versioned/<release>/**` |
| `backend-api`     | `/docs/**`  | `docs/services/backend-api/synced/**` | `docs/services/backend-api/versioned/<release>/**` |
| `scraper-api`     | `/docs/**`  | `docs/services/scraper-api/synced/**` | `docs/services/scraper-api/versioned/<release>/**` |
| `model-api`       | `/docs/**`  | `docs/services/model-api/synced/**`   | `docs/services/model-api/versioned/<release>/**`   |

### Mapping Rules

- only files inside the service repo `/docs/` tree are eligible for automatic sync
- stable central paths are derived from the service alias, not from the source repository display name
- service landing pages in this repository are never overwritten by sync jobs
- `synced/` represents the current latest state for a service
- `versioned/<release>/` stores immutable release snapshots when release sync is enabled

## Artifact and Metadata Standard

Each sync run should publish a structured artifact bundle from the source repository.

### Required Bundle Structure

```text
sync-bundle/
  manifest.json
  docs/
  assets/
```

### Required Manifest Fields

| Field          | Purpose                                      |
| -------------- | -------------------------------------------- |
| `service`      | Stable service alias such as `backend-api`   |
| `source_repo`  | Actual source repository name                |
| `source_ref`   | Source branch, tag, or release ref           |
| `source_sha`   | Commit SHA used to generate the bundle       |
| `generated_at` | Bundle generation timestamp                  |
| `sync_mode`    | `merge`, `release`, or `scheduled-reconcile` |
| `doc_count`    | Number of synced docs in the bundle          |
| `asset_count`  | Number of assets in the bundle               |

### Synced Page Metadata Rules

- required metadata from earlier milestones must remain intact
- synced pages must retain `source_repo` and `source_path`
- release snapshots should also include `source_ref` or an equivalent release identifier in frontmatter or manifest metadata

## Asset and Generated Reference Handling

### Asset Rules

- copy only assets referenced by synced docs or explicitly declared in the sync bundle
- preserve relative relationships between docs and assets where possible
- rename conflicting or generic asset names during bundle generation before central publish

### Generated Reference Rules

- generated docs such as API references or schema pages should be published into a clearly named subtree inside the service sync target
- generated references should be identified explicitly in metadata or manifest data so reviewers know they were machine-produced

### Versioning Rules For Assets

- latest assets live with the `synced/` content
- release assets live with the matching `versioned/<release>/` content
- assets should not be shared across release snapshots unless the pipeline guarantees immutability

## Validation Checks

Every sync pipeline should validate the bundle before the central repository accepts it.

### Required Checks

- source service alias is recognized
- manifest fields are present and valid
- synced docs contain required metadata
- target paths stay inside the allowed service subtree
- no landing page outside `synced/` or `versioned/` is overwritten
- asset references resolve correctly after mapping
- no duplicate target paths are produced inside the same bundle
- central broken-link and build validation still pass after applying the bundle

## Conflict Handling

Conflicts should be handled deterministically.

### Path Conflicts

- if two services attempt to publish to the same central target path, fail the sync and require path correction
- central landing pages always win over service bundles

### Asset Conflicts

- conflicting generic asset names should be renamed in the source bundle generation step
- if the conflict survives into the central sync step, fail validation instead of overwriting silently

### Metadata Conflicts

- if frontmatter is missing required source attribution, reject the bundle
- if service alias and source metadata disagree, reject the bundle

## Rollback Strategy

Documentation sync should behave like a controlled publish, not an uncontrolled overwrite.

### Rollback Rules

- apply sync changes atomically in a dedicated commit or publish step
- keep the previous successful central state as the rollback target
- if central validation fails, do not publish the new bundle
- if a published sync is later found to be bad, revert the sync commit or restore the last known good bundle

### Last-Known-Good Principle

The central repository should always be recoverable to the previous valid documentation state without requiring the service team to regenerate the original bundle immediately.

## Versioning Strategy

Bisakerja should separate **latest** documentation from **release snapshot** documentation.

### Latest Track

- `synced/` always reflects the latest accepted service documentation
- latest is the default reading path for most users

### Release Track

- `versioned/<release>/` stores immutable snapshots for tagged or release-driven syncs
- release snapshots are useful when the central docs must explain behavior tied to a specific shipped service version

### When To Use Release Snapshots

- service contracts or behavior differ materially by release
- incident review needs historical documentation context
- regulated or stakeholder-reviewed releases require stable docs evidence

## Changelog and Release-Note Behavior

The central repository should not generate verbose changelogs for every sync by default. It should record high-signal release behavior instead.

### Merge Sync Behavior

- use commit messages or PR summaries for routine latest-doc updates
- avoid noisy changelog generation for every small sync

### Release Sync Behavior

- create a concise release-note summary listing changed services and release snapshot identifiers
- link the release note to the corresponding `versioned/<release>/` paths

### Scheduled Reconciliation Behavior

- scheduled reconciliation should produce drift or health reports, not reader-facing release notes

## Approval Boundary

This document defines the technical design for synchronization and versioning. It does **not** itself imply organizational approval to deploy the pipeline. Formal approval should happen through the team’s normal engineering review process.
