# Scaling and Extension Strategy

This document defines how the documentation platform should scale beyond the current four-service model.

## Scaling Principles

- New capabilities should fit the existing structure before creating new top-level sections
- Service onboarding should be repeatable and low-friction
- Future extension points should preserve stable routes and ownership boundaries
- Automation should reduce manual coordination, not hide responsibility

## Adding A New Service

New services should be onboarded through a consistent checklist.

### Required Steps

1. Assign a stable service alias in kebab-case.
2. Create a central landing page under `docs/services/<service>/index.mdx`.
3. Reserve a `docs/services/<service>/synced/` subtree for latest synced docs.
4. Register the service in sync and versioning configuration.
5. Add ownership metadata and the serviceâ€™s required reviewers.
6. Add the service to central navigation and the service index.
7. Validate that the service can produce the minimum content package defined in the service documentation integration standard.

### Optional Step

- Enable `versioned/<release>/` snapshots if the service has meaningful release-specific behavior or contracts

## When To Add New Top-Level Sections

Do not add a new top-level section unless:

- The content cannot fit naturally into `overview/`, `services/`, `operations/`, `standards/`, or `references/`
- There is an ongoing body of content with its own navigation needs
- Ownership and lifecycle rules are clear

## Future Documentation Types

The current model should be able to grow to include:

- Service release snapshots
- Generated API or schema references
- Architecture decision records
- Runbooks
- Incident retrospectives
- Compliance or evidence documentation

### Preferred Placement

| Content Type                 | Preferred Placement                                                               |
| ---------------------------- | --------------------------------------------------------------------------------- |
| Service release snapshots    | `docs/services/<service>/versioned/<release>/`                                    |
| Generated API or schema docs | `docs/services/<service>/synced/generated/` or release-specific equivalent        |
| ADRs                         | `docs/references/adr/` or another clearly scoped reference subtree                |
| Runbooks                     | `docs/operations/runbooks/`                                                       |
| Incident retrospectives      | `docs/operations/retrospectives/`                                                 |
| Compliance docs              | `docs/operations/compliance/` or a future dedicated section if scale justifies it |

## Extension Rule

Every new documentation type should first answer:

- Who owns it
- How it is reviewed
- How it becomes stale
- Whether it is synchronized, authored centrally, or generated
