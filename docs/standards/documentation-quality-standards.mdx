# Documentation Quality Standards

This document defines the minimum quality bar for documentation published in the Bisakerja documentation hub.

## Quality Principles

- documentation should help a reader act, not just exist
- scope should be clear enough that a reader can tell what a page owns and what it does not
- documents should optimize for correctness before completeness theater
- duplicated content should be minimized in favor of stable links to the canonical page

## Required Quality Dimensions

| Dimension              | Standard                                                        |
| ---------------------- | --------------------------------------------------------------- |
| Clarity                | The page states its purpose and audience clearly                |
| Structure              | The page uses predictable sections and headings                 |
| Technical completeness | The page covers the minimum content expected for its type       |
| Ownership              | The page includes owner and review metadata                     |
| Link integrity         | The page does not introduce broken links or orphaned navigation |
| Maintainability        | The page avoids unnecessary duplication and unstable references |

## Writing Rules

- prefer direct, specific statements over vague summary language
- describe system behavior in terms of responsibilities, flows, and boundaries
- call out assumptions when a detail is inferred rather than confirmed
- avoid copy-pasting service-owned detail into central pages when a stable link is enough

## Technical Completeness By Page Type

### Overview Pages

- explain platform intent or cross-service behavior
- link to relevant services and references
- state known assumptions when details are not yet synchronized from source repos

### Service Landing Pages

- describe service purpose and platform responsibility
- identify sync target and ownership boundaries
- list expected service-owned documentation and current gaps

### Operational Pages

- identify the affected environments or services
- explain expected workflows or failure states
- provide a useful debugging or decision-making frame

### Standards Pages

- define explicit rules, not only guidance
- include enough structure that contributors can follow the rule without asking for interpretation

## Quality Failure Examples

- a page has no identifiable owner
- a service page duplicates implementation detail that is supposed to live in the source repository
- a standards page gives advice without defining enforceable rules
- a document is technically correct but impossible to navigate to from the site
