# Archival Strategy

This document defines how old or superseded documentation should be retired without losing useful historical context.

## When To Archive

Archive a document when:

- A service is deprecated or replaced
- A major architecture pattern is no longer current
- A user flow is no longer supported
- Historical evidence is useful, but the page should not guide new work

## Archive States

- Use `deprecated` when readers may still need the page temporarily
- Use `archived` when the page is for history only

## Archive Placement

Preferred archive models:

- Keep archived service material under the service subtree with clear lifecycle metadata
- Move cross-service historical material into a clearly named archival subtree only if volume justifies it

## Archive Rules

- Archived pages should not appear as the primary recommended path in onboarding or landing pages
- Archived pages should explain what replaced them
- Archived pages should preserve enough metadata to support incident review, audits, or migration work

## What Not To Archive Silently

- Architecture decisions that still influence the present system
- Runbooks still used in current operations
- Compliance evidence still required for active controls
