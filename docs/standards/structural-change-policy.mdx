# Structural Change Policy

This document governs changes to navigation, directory structure, stable routes, and document placement in the central docs repository.

## Structural Changes Covered

- Sidebar or navbar changes
- Moving a page to a different section
- Changing stable service sync targets
- Renaming canonical docs paths
- Reorganizing top-level documentation sections

## Rules

- Stable reader-facing routes should not change without a migration plan
- Structural changes must be reviewed as documentation architecture changes, not only content edits
- Service sync targets should remain stable unless there is a compelling platform-level reason to change them

## Change Procedure

1. Identify the reason for the structural change
2. List affected paths and navigation surfaces
3. Update related standards or landing pages
4. Validate the full docs site
5. Communicate the structural change in the relevant PR or release note

## Breaking Change Rule

If a structural change would break expected reading paths or synchronization assumptions, it should be treated as a breaking documentation change and reviewed accordingly.
