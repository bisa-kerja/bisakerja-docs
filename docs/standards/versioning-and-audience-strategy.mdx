# Versioning and Audience Strategy

This document extends the current sync design into a broader future strategy for versioned docs, localization, and audience-specific documentation views.

## Broader Versioned Docs Support

The current synchronization and versioning model already defines service release snapshots. This document extends that into a longer-term platform strategy.

### Use Cases For Versioned Docs

- API behavior differs by release
- Service contracts change materially over time
- Support or incident review requires historical documentation context
- Stakeholder review needs a stable release-aligned docs set

### Recommended Model

- Keep `synced/` as the latest track
- Use `versioned/<release>/` for release-specific snapshots
- Avoid versioning all content by default; version only where the maintenance cost is justified

### Scope Rule

- Version service-local docs first
- Version central overview docs only if platform behavior is meaningfully release-coupled

## Localization Strategy

Localization should be added only when there is a real audience need and a sustainable review process.

### Preconditions

- Stable source language ownership
- Translation review responsibility
- Clear decision on which sections must be localized

### Recommended Priority

1. Onboarding and start-here pages
2. Product-facing overview content
3. Selected service or operational content as needed

### Non-Goal

- Do not localize every low-level technical page by default if freshness cannot be maintained

## Audience-Specific Views

Not every audience needs the same reading path. Future enhancements may include:

- Stakeholder-focused entry collections
- Engineer-focused deep-dive collections
- Support or operations-focused collections

The existing role-oriented guides are the first form of this strategy and should be extended rather than replaced.

## Generated Reference Strategy

Generated references are appropriate when manual maintenance is less reliable than source-driven generation.

### Good Candidates

- API endpoint references
- Schema references
- Model contract summaries
- Interface catalogs

### Rules

- Generated docs must be clearly labeled as generated
- Generated docs should live in a predictable subtree
- Generated docs must still satisfy metadata and validation requirements
- Human-authored orientation pages should introduce generated reference sets instead of exposing them without context
